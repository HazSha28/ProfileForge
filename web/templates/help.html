ProfileForge Documentation

ProfileForge merges heterogeneous candidate data from multiple sources into a single, validated, canonical JSON profile — with full provenance tracking and deterministic confidence scoring on every field.

📥
Input Recruiter CSV + Resume PDF + optional platform links
⚙️
Pipeline Parse → Normalise → Merge → Score → Validate → Project
📤
Output Structured JSON with sources and confidence per field

Quick Start

Follow these three steps to generate your first canonical profile.

  1. 1
    Upload a Recruiter CSV

    Prepare a CSV file with candidate details. The column names are flexible — see the CSV Template section for the supported column names.

  2. 2
    Upload a Resume PDF

    Upload the candidate's resume as a PDF. ProfileForge uses pattern recognition to extract name, contact details, skills, and work experience from unstructured text.

  3. 3
    Add Platform Links (Optional)

    Expand the Candidate Platform Links section and paste any known profile URLs — LinkedIn, GitHub, portfolio, LeetCode, and more.

  4. 4
    Click Transform Profile

    The pipeline runs in seconds. Review the canonical profile on screen, then download the JSON or copy it directly.

Resume PDF

The PDF is the unstructured source. ProfileForge extracts text from every page and uses pattern matching to locate fields.

FieldExtraction MethodNotes
NameFirst non-contact line heuristicWorks best when name is on line 1
EmailRFC 5321 regexAll emails found are captured
PhoneInternational phone regexValidated via libphonenumber
LinkedIn / GitHubURL pattern matchingFirst match per platform
SkillsSkills section detectionComma, pipe, bullet delimiters

Years ExperienceDerived from date range spanFalls back to "X years of experience" pattern
Tip: PDF extraction quality depends on the resume layout. Text-based PDFs work best. Scanned image PDFs are not supported.

Before merging, every field is normalised to a canonical form.

EmailLowercaseJane@EXAMPLE.com → jane@example.com
NameTitle caseJOHN DOE → John Doe

SkillsAlias map to canonical nameJS → JavaScript
DatesISO 8601 (YYYY-MM-DD)Jan 2022 → 2022-01-01

Merge Policy

When both sources provide a value for the same field, a deterministic policy decides the winner. There is no random behaviour.

Field TypeRule
Scalar (name, headline, years) Resume wins over CSV. If timestamps are available, the newest value wins regardless of source.
Lists (emails, phones, skills) Deduplicated union of all sources. Order follows source priority (Resume first).
Nested (location, links) Each sub-field merged independently using the scalar rule.
Priority order: Resume > CSV > LinkedIn (if added in future)

Confidence Scores

Every field carries a confidence score from 0.0 to 1.0 based on how many sources contributed to it.

SourcesScoreMeaning
Resume only0.6Candidate-authored, single source
CSV only0.5Recruiter-supplied, single source
Resume + CSV0.85Cross-verified by two sources
Resume + CSV + LinkedIn1.0Triple-verified, fully trusted

Provenance

Every field in the output records which source(s) it came from in a sources array.

{
  "skills": {
    "value": ["Python", "JavaScript", "Docker"],
    "sources": ["Resume", "CSV"],
    "confidence": 0.85
  }
}

Output Schema

The canonical profile JSON has the following top-level structure. Each field is wrapped with value, sources, and confidence.

{
  "candidate_id": "uuid-v4",
  "full_name":        { "value": "Jane Doe",      "sources": ["Resume","CSV"], "confidence": 0.85 },
  "emails":           { "value": ["jane@..."],     "sources": ["Resume"],      "confidence": 0.6  },
  "phones":           { "value": ["+14155552671"], "sources": ["CSV"],         "confidence": 0.5  },
  "location":         { "value": { "city": "San Francisco", "region": "CA", "country": "US" }, ... },
  "links":            { "value": { "linkedin": "...", "github": "..." }, ... },
  "headline":         { "value": "Senior Software Engineer", ... },
  "years_experience": { "value": 5, ... },
  "skills":           { "value": ["Python", "JavaScript"], ... },
  "experience":       { "value": [ { "title": "...", "company": "...", "start_date": "...", "end_date": "..." } ], ... }
}

FAQ

What happens if my CSV uses different column names?

ProfileForge uses a flexible alias map. Common variations like mobile, telephone, e-mail, and state are all recognised. If your column still isn't matched, check the CSV Template table above.

What encoding should my CSV be saved in?

Any encoding works. ProfileForge automatically tries UTF-8, UTF-8 with BOM, Windows-1252 (cp1252), and Latin-1 — so files exported from Excel are handled correctly.

Does the PDF parser work with scanned resumes?

No. Scanned PDFs are images and contain no extractable text. The resume must be a text-based PDF (the kind you can highlight and copy text from).

Why is a field showing confidence 0.5 instead of 0.85?

Confidence 0.5 means the field came from the CSV only. If the same field was also found on the resume, it would score 0.85. Check that the resume PDF contains the relevant information in a parseable format.

Can I add my own skill aliases?

Yes. Open src/normalizers/skills.py and add entries to the SKILL_ALIASES dictionary. The key is any lowercase alias, the value is the canonical display name.

How do I control which fields appear in the output?

Paste a config JSON in the Advanced section. Use include to whitelist fields, exclude to remove fields, and rename to change key names. See the Config File section for examples.
